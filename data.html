2.2 Key Insight: Self-Correcting Pipeline

This run demonstrated AAICLAW's ability to detect and correct label quality issues mid-pipeline. The system:

  1. Detected that raw ODPS labels contradicted the IDV Bypass Report ground truth
  2. Automatically created a relabeling script to correct the issue
  3. Merged corrected data with existing training data
  4. Retrained and achieved 100% accuracy on the corrected test set

A human engineer would have spent 4-8 hours diagnosing this label mismatch. The system identified and resolved it within the same pipeline run.

Date: 2026-03-13
Test set: alert_20260310 (91 attack/alert images)
Baseline model: run_20260313_161500 (epoch08, mini data)
Fine-tuned model: run_20260313_finetune (epoch06, mini + alert)

Test Set Overview:
- valid: 83 images (genuine documents)
- edited: 7 images (manually edited/tampered)
- fake_card: 1 image
- Total: 91 (some corrupted, ~86 participated in evaluation)
- Label mapping: valid → 0, edited/fake_card → 1 (attack)

Overall Metrics:
| Metric              | Baseline | Fine-tuned | Change     |
|---------------------|----------|------------|------------|
| Accuracy            | 9.3%     | 69.8%      | +60.5 pp   |
| Valid Correct        | 1/78     | 59/78      | +74.4 pp   |
| Attack Correct       | 7/8      | 1/8        | -75.0 pp   |
| Predictions Changed  | -        | 64         | -          |
| Improved             | -        | 58         | -          |
| Regressed            | -        | 6          | -          |

Behavior Analysis:
- Baseline: Almost all predictions were "attack" (pred=1), strong bias.
  Valid recall extremely low (1/78), massive false positives.
- Fine-tuned: Valid class significantly improved (59/78 correct).
  Attack class severely degraded (1/8 correct).
  Model shifted from "biased toward attack" to "biased toward valid".

Root Cause: Joint training had valid samples dominating
(333 valid vs 258 photoshop + 250 genai).

4.2 Ensembler Architecture

The forgery defense system uses a two-level architecture:

Level 1: Sub-model inference
  Each image is scored by all 7 sub-models independently.
  Output: 7 float scores (0-1) per image.

Level 2: LightGBM ensembler
  Takes 7 sub-model scores as features.
  Trained per-country with country-specific thresholds.
  Output: final forgery/genuine decision per card type per country.

Countries with dedicated thresholds:
  IDN (Indonesia), PHL (Philippines), THA (Thailand),
  MEX (Mexico), MYS (Malaysia), World (all others)

Threshold calibration:
  Per-country, per-card-type JSON files defining
  FAR/FRR operating points. Human-approved at Gate 3.

5. Full Pipeline Specification (8 Stages)

Stage 1 — Monthly Data Collection (Skill: monthly-data-collection)
  Scripts: fetch_solution.py, fetch_annotation.py, merge_label_balance.py,
           download_images.py, validate_images.py
  Output: solution_{month}_global_train_validated.csv + images

Stage 2 — Preprocessing: UCC + Card Alignment
  Script: data_run.sh → ucc_service.py + card_alignment.py
  Adds: pts, card_type, rotation, card_bbox columns
  Filter: pts == 'nopts' → no card detected → filtered out

Stage 3 — Data Routing
  Notebook: data_routing.ipynb
  Routes monthly batches to Benchmark Pool or Training Pool
  Dedup by path then card_bbox, normalize country codes
  Benchmark: per-country CSVs (IDN, PHL, THA, MEX, MYS, World)
  Training: global_train_v5_ucc.csv

Stage 4 — Per-Model Dataset Preparation
  Category A (from data routing): Photocopy, SCX, Face Tampering, Text Tampering
  Category B (dedicated annotation): Material, Passport, GenAI
  Each model has specific label mappings, balance ratios, and required columns

Stage 5 — Model Training
  Framework: tdd.train.main (PyTorch Lightning)
  Commands: lmdb (cache) → fit (train)
  Checkpoints: epoch{N}-step{N}-loss{N}-acc{N}.ckpt

Stage 6 — Inference on Benchmark Data
  Scripts: run_benchmark.sh → run_model.sh
  Runs trained model on benchmark CSVs → rst.csv

Stage 7 — Benchmark Analysis
  Per-model benchmark notebooks
  Compute FAR/FRR, per-country metrics, threshold analysis

Stage 8 — ONNX Export
  Script: release_onnx.sh
  Only after benchmark confirms improvement over production model

7. Compound Value — Experience Memory

The Experience Memory system captures structured knowledge from every pipeline run.
Each entry records: what worked, what failed, why, and what to try next time.

Value accumulation curve:
  Run 1-5   (Month 1-2):   ~30% time saved — automation value only
  Run 5-15  (Month 3-6):   ~60% time saved — experience kicks in
  Run 15+   (Month 6+):    ~80% time saved — institutional expert

Knowledge categories tracked:
  - Attack type effectiveness: which approaches work for which forgery types
  - Regional quirks: template changes, document quality variations per country
  - Annotation pitfalls: common labeling errors and how to detect them
  - Threshold sensitivity: per card type score distribution patterns
  - Training strategies: class weighting, data augmentation, checkpoint selection
